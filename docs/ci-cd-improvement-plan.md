# CI/CD improvement plan

## Summary

Autorota's CI/CD is unusually complete for a project this size: a PR/push gate (`ci.yml`), automated versioning + changelog (`release-plz.yml`), a tag-triggered release pipeline that signs and ships to TestFlight for iOS and macOS and produces a notarized `.dmg` (`release.yml`), weekly supply-chain scanning, PR-title linting, and Dependabot. The design is sound.

The catch: as of this writing, **the pipeline has never actually run for a real release.** There are no git tags in the repo, `CHANGELOG.md` has only an `[Unreleased]` section, `Cargo.toml` is at `0.1.0`, and the Xcode project's `MARKETING_VERSION` / `CURRENT_PROJECT_VERSION` are still the untouched Xcode defaults (`1.0` / `1`). Meanwhile App Store submission material (`appstore-copy.md`, `design/appstore/` screenshots) is being prepared right now. So the priority isn't "build CI from scratch" — it's **de-risking the first real run of a pipeline that has only ever been tested by reading**, plus closing a handful of concrete gaps that only matter once the app is a *live* App Store product (crash symbolication, pipeline-failure alerting, confirming an upload actually became a usable build).

Gaps below are grouped by when they should be addressed relative to the first real tag push, not by abstract severity — that ordering is the point.

## Gap table

| # | Gap | Why it matters for a released app | Fix | Effort |
|---|---|---|---|---|
| 1 | `ci.yml`, `release.yml`, `perf.yml` pin `runs-on: macos-15` and select Xcode 26.0 via `setup-xcode`. GitHub's `macos-26` runner image went GA 2026-02-26; `macos-15` images have documented storage constraints that block a full Xcode 26.x install. `GAPS.md` already flags this as "Critical — CI will fail on Swift jobs" but has no prescribed fix. | Every Swift job (ViewModel tests, compile checks, SPM integration, both release archive jobs) is at risk of failing on the exact runner it's pinned to, right as it's about to be exercised for real. | Change `runs-on: macos-15` → `runs-on: macos-26` in all four Swift/Apple jobs' `runs-on` lines across `ci.yml`, `release.yml`, and `perf.yml`. Push a no-op PR afterward and confirm the full matrix goes green before relying on it. | S |
| 2 | The release pipeline (`release.yml`) has zero real executions. Nobody has seen it succeed end to end — signing, `-allowProvisioningUpdates`, TestFlight upload, notarization, `.dmg` staple, GitHub Release — against this exact repo, these exact secrets, this exact Apple Developer account state. | Any of a dozen small things (an unrenewed cert, a wrong team ID, a missing secret, a typo in an `ExportOptions*.plist`) will surface for the first time on the first tag — i.e. under the most time pressure, the day you're trying to actually ship. | Before cutting the real first version, push a throwaway tag below the real version (e.g. `v0.0.1` — note a `-rc1` suffix will NOT work: the workflow's tag glob and the `tag-parse` validation both require exactly `vX.Y.Z`) and watch `release.yml` run to completion. Fix whatever breaks. Delete the TestFlight build / GitHub Release afterward if desired — the point is exercising the pipeline once at low stakes. | M |
| 3 | No `.dSYM` capture. `release-ios` and `release-macos` archive and export but never collect or upload debug symbols; nothing in the repo references Sentry, Crashlytics, or any symbolication service. | Crash reports from real users on a live app are close to unreadable without matching `.dSYM`s. Xcode archives retain them locally, but CI-built archives disappear when the runner is torn down — once shipped, there is currently no way to symbolicate a crash from a CI-built binary. | Add a step in both `release-ios` and `release-macos` that zips `$RUNNER_TEMP/*.xcarchive/dSYMs` and uploads it as a build artifact (`actions/upload-artifact`, e.g. 90-day retention) at minimum. If/when a crash reporter is adopted, wire the same step to upload directly to it instead. | S |
| 4 | No failure notification on any workflow. A `release.yml` failure partway through (e.g. iOS ships, macOS fails) is currently visible only as a red check in the Actions tab. | A release that's half-shipped and silently stuck is the worst state for a live app's pipeline to be in — nobody finds out until someone checks GitHub by hand. | Add a final `if: failure()` step to `release.yml` (and optionally `ci.yml` on `main`) posting to whatever channel the team actually watches — Slack webhook, email via an action, or at minimum a GitHub Issue auto-filed via `actions/github-script`. Pick the lightest option that will actually get read. | S |
| 5 | `upload-testflight-build` succeeding only confirms the HTTP upload succeeded. Apple's binary processing (export compliance, Info.plist validation, asset validation) happens asynchronously afterward and can fail with zero signal back to the workflow. | "Green CI" currently does not guarantee a build became usable in TestFlight. First time this bites will be confusing — CI says success, App Store Connect shows nothing. | Document this explicitly (done in the consolidated guide) as a required manual check for now: after `release.yml` goes green, confirm the build shows "Ready to Test" in App Store Connect before telling testers. Automating a poll against the App Store Connect API is a reasonable follow-up once the manual step has been done a few times and the friction is felt. | S (doc) / M (automate later) |
| 6 | `.gitignore` excludes `platforms/apple/ExportOptions*.plist` ("contain signing identities / team IDs") but all three plists are tracked in the working tree right now. | Landmine either way: someone will eventually trust `git status` being clean when regenerating these and be wrong, or assume they're safely untracked when they aren't. Team ID and export method aren't secret, but the intent behind the ignore rule and the actual repo state currently disagree. | Pick one and make the repo consistent: (a) remove the `.gitignore` entry and add a one-line comment above the plists explaining they're intentionally committed (they don't contain private keys, just team ID + export method), or (b) actually `git rm --cached` them and document the regeneration steps. (a) is less churn given they're already tracked and referenced directly by path in `release.yml`. | S |
| 7 | `platforms/apple/Apps/AutorotaApp/ci_scripts/ci_post_clone.sh` is a live Xcode Cloud hook (Apple auto-detects `ci_scripts/`), last touched 2026-05-11, undocumented in any current doc, and functionally superseded by `release.yml`. | If an Xcode Cloud workflow is still enabled in App Store Connect for this app, every push to a watched branch triggers a second, unmonitored build/submit path running in parallel with GitHub Actions — possible duplicate TestFlight builds or conflicting build numbers. | Check App Store Connect → Xcode Cloud for this app. If no workflow is configured there, the `ci_scripts/` hook is inert — leave it (it's cheap insurance if Xcode Cloud is ever turned on) but add a one-line comment noting it's currently unused and why it exists (the two DNS/PATH incidents in `BUG_LOG.md`). If a workflow *is* configured, either disable it or explicitly document why two release paths coexist. | S |
| 8 | No root `README.md`, so no CI status badges or landing-page context for the repo. | Minor, but a shipped app's repo having zero visible build status at a glance is an easy, cheap fix. | Add a short root `README.md` with build status badges for `ci.yml` and `release.yml` and a link to `docs/ci-cd-guide.md`. | S |
| 9 | Tauri desktop (`app-desktop`) has zero packaging/signing/publishing automation — `ci.yml` only runs `cargo check` on it. | Not a regression — desktop was never wired for release — but worth naming explicitly so it reads as a deliberate deferral rather than something that got missed. | No action unless/until desktop distribution becomes a near-term goal. Tracked here so it doesn't get "rediscovered" as a surprise gap later. | — (deferred) |
| 10 | No automation pushing `appstore-copy.md` / `design/appstore/` screenshots to App Store Connect — metadata updates are manual today. | Nice-to-have, not a launch blocker. Manual App Store Connect editing is fine for an infrequent first submission. | Revisit after the first manual submission, once it's clear how often metadata actually changes. If it's frequent, `fastlane deliver` or the App Store Connect API is the standard tool. | — (deferred) |

Not a gap, confirmed correct as-is: `INFOPLIST_KEY_ITSAppUsesNonExemptEncryption = NO` is already set, so export-compliance prompts won't block submission. `perf.yml`'s regression gate is intentionally `continue-on-error: true` (soft gate, by design per its own header comment) — not a bug, just noting it here so it isn't "fixed" into a hard gate by accident later.

## Suggested sequencing

1. **Before the first real tag** (cheap, removes real risk): #1 (runner migration), #2 (dry-run release on a `-rc1` tag), #6 (ExportOptions/gitignore reconciliation — five-minute fix, do it while already touching release config).
2. **Ship the first real version**, manually supervised — watch `release.yml` run live, manually confirm the TestFlight build reaches "Ready to Test" in App Store Connect (#5, until automated).
3. **Before the second release**: #3 (dSYM capture), #4 (failure notification), #7 (confirm Xcode Cloud status).
4. **As time allows**: #8 (README badges).
5. **Deferred, revisit only if the underlying need shows up**: #9, #10.
